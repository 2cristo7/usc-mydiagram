6 tipos de diagrama

agent/schemas.py · DiagramType enum · ALLOWED_NODE_TYPES / ALLOWED_EDGE_TYPES

El agente soporta seis tipologías de diagrama de software, cada una con su vocabulario propio de nodos y aristas (registrado en ALLOWED_NODE_TYPES / ALLOWED_EDGE_TYPES en agent/schemas.py). El usuario describe en lenguaje natural qué quiere; el nodo classify del grafo LangGraph elige el tipo (o respeta la preselección del usuario) y el extractor usa el prompt específico de ese tipo para producir la topología correcta.

erd

Entidad-Relación

Tablas con columnas tipadas, claves primarias y relaciones por cardinalidad. Nodo table con cabecera + filas. Permisivo en validación estructural (tablas de catálogo sin relaciones son legítimas).

TABLE ONE_TO_ONE ONE_TO_MANY MANY_TO_MANY
flowchart

Diagrama de Flujo

Pasos, decisiones y terminadores. Validación estructural activa: debe existir un terminator de inicio y uno de fin, con el grafo conexo. El rombo de decisión despliega ramas condicionales etiquetadas.

TERMINATOR STEP DECISION FLOW CONDITIONAL
sequence

Diagrama de Secuencia

Actores con línea de vida y activaciones derivadas del layout. Mensajes y respuestas con etiquetas editables; fragmentos combinados (alt, opt, loop, par) en un array fragments. Validación activa: al menos dos actores que interactúen.

ACTOR SEQUENCE SEQUENCE_REPLY
use_case

Casos de Uso (UML)

Actores, casos de uso y subsistemas. Relaciones de asociación, inclusión, extensión y generalización. Nodos actor, use_case y system. Permisivo en validación estructural.

ASSOCIATION INCLUDE EXTEND INHERITS
mindmap

Mapa Mental

Un único nodo topic que adopta tres roles según su posición (raíz, rama u hoja), con ramas jerárquicas en layout radial. Edge association con curvas orgánicas. Validación activa: debe existir un topic raíz.

TOPIC ASSOCIATION
architecture

Arquitectura (C4)

Notación C4 (person, system, container, component) más service, database, queue y gateway. Admite grupos y contenedores redimensionables. Permisivo: un servicio sin cablear es un diseño válido.

SYSTEM CONTAINER SERVICE DATABASE CALLS DEPENDS_ON
Decisión cerrada

Los prompts de extracción son específicos por DiagramType (registrados en agent/prompts.py). Un prompt genérico con ejemplo ERD producía topologías lineales para todos los tipos. El mapa de tipos permitidos en agent/schemas.py es la única fuente de verdad semántica del agente.

Generación conversacional

frontend/src/components/ChatPanel.tsx · FloatingPrompt.tsx · ChatMessage.tsx

Entrada libre en lenguaje natural

El usuario describe el diagrama que quiere en el panel de chat (ChatPanel) o mediante el FloatingPrompt. No se requiere sintaxis especial ni schema previo.

El mensaje viaja vía Socket.io al backend Node, que aplica rate limit y consulta la caché antes de delegar al agente Python con FastAPI.

Socket.io generate refine

Streaming progresivo al canvas

El agente emite nodos y aristas confirmados (streaming NDJSON) a medida que los valida. El canvas los renderiza incrementalmente — solo se añade, nunca se retracta.

Al terminar, el evento done lleva el snapshot completo y se aplica incondicionalmente (reconciliación), garantizando coherencia aunque se pierda algún evento de stream.

NDJSON streaming Reconciliación

Refinamiento iterativo (agente ReAct)

Tras la generación, el usuario puede pedir cambios en lenguaje natural. El agente ReAct (LangGraph) dispone de nueve herramientas incrementales: find_node, add_node, update_node, delete_node, add_edge, delete_edge, apply_layout, ask_clarification y regenerate_from_scratch. El loop termina cuando el LLM deja de emitir tool calls.

Si el agente necesita aclaración, emite ask_clarification, que suspende el grafo con un interrupt() real (LangGraph). El usuario responde y el agente reanuda desde el mismo punto.

Edición inline tipo Miro

frontend/src/components/edges/ · nodes/ · EditToolbar.tsx · NodePropertiesPanel.tsx

Una vez generado, el diagrama es completamente editable sin salir de la vista del canvas. El modelo de interacción toma como referencia Miro: doble clic para editar texto inline, arrastrar para mover, handles para reconectar aristas.

Exportación

frontend/src/components/ExportMenu.tsx

Export PNG

Usa la API toBlob() de React Flow para capturar el canvas completo con el viewport actual. El nombre del archivo se deriva del título del diagrama.

Disponible toBlob()

Formato nativo e interoperabilidad

El formato nativo .mdia serializa el diagrama completo como DiagramSchema (validado con Zod, diagramImportSchema) y admite reimportación exacta con integridad referencial (sin aristas huérfanas). Además exporta e importa Mermaid, draw.io y Excalidraw.

Disponible .mdia Mermaid draw.io Excalidraw
Fuente de verdad El import nativo valida con diagramImportSchema = diagramSchema + .refine de integridad referencial en runtime (Zod). Los tipos TS se derivan de los schemas Zod (z.infer), no al revés: el schema es la única fuente de verdad.

Autenticación y persistencia

frontend/src/components/AuthButton.tsx · backend/src/diagrams.ts

El acceso está protegido con Google OAuth gestionado por Supabase Auth. El backend Node verifica el JWT firmado por Supabase y extrae el user_id (claim sub) antes de cualquier operación. El frontend nunca accede directamente a Supabase: todo pasa por el backend.

Componente Responsabilidad
AuthButton.tsx Botón de login/logout con Google. Muestra avatar y nombre del usuario autenticado.
Supabase Auth Gestiona el flujo OAuth con Google y emite el JWT de sesión. No se monta servidor OAuth propio.
backend/src/diagrams.ts CRUD de diagramas. El backend reenvía el JWT del usuario a Supabase (cliente por petición con Authorization: Bearer) para que la RLS imponga la propiedad.
RLS en tabla diagrams Políticas Row Level Security: un usuario solo puede leer/escribir sus propios diagramas (auth.uid() = user_id).
Auto-guardado POST al crear / PATCH con debounce al editar. Las coordenadas de los nodos se incluyen en el guardado.
Revalidación JWT en socket Lazy: cada llamada al agente compara now ≥ exp. Un token caducado provoca auth:expired + disconnect. El cliente refresca con auth:refresh.
Decisión cerrada

El backend es el único punto de acceso a datos de persistencia. El frontend nunca llama directamente a Supabase. La service_role queda reservada para la caché global (generation_cache), que no tiene user_id y no debe ser accesible desde el cliente.

Caché de generaciones

backend/src/ · tabla generation_cache

Para prompts repetidos, el backend sirve el diagrama generado previamente sin invocar el LLM, reduciendo latencia y coste. La caché es global (no por usuario), gestionada exclusivamente por el backend con service_role.

AspectoDiseño
Clave (prompt_key, model) — prompt normalizado (trim + minúsculas + espacios) + LLM_PROFILE
Qué se cachea Solo generación (no refinamiento). Solo éxito limpio: ≥ 1 nodo, sin degradaciones.
Invalidación upsert — solo el último diagrama por clave, renovando el TTL. El botón "Regenerar" del usuario fuerza generación nueva saltándose el lookup.
Seguridad RLS activada sin políticas → solo service_role lee/escribe. El cliente no puede envenenar la caché. checkRateLimit corre antes del lookup: un usuario en el tope no llega a la caché.
Decisión cerrada

El rate limiter se trasladó del agente Python al backend Node (keyed por user_id o IP de socket). El agente Python veía siempre la IP del backend — una cubeta global inútil. Ahora el backend ve la identidad real del usuario.

🚦

Rate Limit

backend/src/rateLimit.ts

El control de admisión de peticiones al LLM vive en el backend Node, en backend/src/rateLimit.ts. Se aplica a las tres entradas socket que corren el agente (generate, refine, refine/resume).

🔑

Keyed por identidad real

Si el usuario está autenticado, la clave es su user_id. Si no, la IP del socket. Esto evita la cubeta global del agente Python.

🔢

Contador por petición

El contador se incrementa por petición, no solo por llamada al LLM. Un hit en caché también consume quota, por diseño.

⛔

Prioridad ante caché

checkRateLimit corre siempre antes del lookup de caché. Un usuario en el tope no accede a la caché ni al LLM.

📢

Error accionable

El evento de rate limit incluye cuántas peticiones quedan y cuándo se resetea la ventana, para que el frontend pueda mostrar un mensaje útil al usuario.

Validación multicapa del agente

agent/ · schemas.py · structural.py

La calidad del diagrama generado está garantizada por tres niveles de validación encadenados dentro del grafo LangGraph, con bucles de reintento independientes por presupuesto:

NivelQué compruebaAcción si fallaPresupuesto
Local — nodos Pydantic (DiagramNode) + tipo en ALLOWED_NODE_TYPES Retroalimentación al extractor con el error; bucle de regeneración de nodos node_retry_count
Local — aristas Pydantic (DiagramEdge) + tipo en ALLOWED_EDGE_TYPES + no huérfana Retroalimentación al extractor; las huérfanas se retienen (no se emiten) retry_count
Estructural Diagrama ensamblado: terminator de inicio y fin (grafo conexo) en flowchart, topic raíz en mindmap, dos actores que interactúen en sequence. Registrada solo para estos tres tipos. Bucle de relleno (extract_nodes o extract_edges); solo AÑADE schema_retry_count
Invariante de streaming Lo que ya se emitió al canvas no se retracta. El reintento opera solo sobre piezas retenidas (nunca streameadas). La reconciliación del evento done garantiza coherencia final.

UX: selección de tipo y estados

frontend/src/components/DiagramTypeCards.tsx · TopBar.tsx · ToolTray.tsx

Preselección de tipo desde la UI

DiagramTypeCards muestra los 6 tipos con su icono y descripción. El usuario puede elegir antes de escribir el prompt. El tipo viaja en el contrato de petición (diagram_type) hasta el agente. El "Automático" no es un tipo: es la ausencia de diagram_type, que deja que el agente clasifique.

El nodo classify en el grafo separa "clasificar tipo" de "generar título": si el tipo llega preseleccionado, se omite la clasificación pero se mantiene la generación del título.

Disponible diagram_type

Estados UX: empty / loading / error

Estados interesantes para cada fase del ciclo de vida del canvas: pantalla de bienvenida cuando no hay diagrama, animación de carga durante generación, y mensaje accionable con causa real en caso de error.

Absorbe deuda de docs/pendientes.md: feedback de error en export PNG, errorMessage real del store, transición error → idle, caída del servidor durante generación.

Disponible TopBar ToolTray
