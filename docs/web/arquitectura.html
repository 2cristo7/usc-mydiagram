ARQUITECTURA DEL SISTEMA

Tres procesos independientes

Frontend, Backend y Agente IA corren como procesos separados que se comunican únicamente por HTTP y WebSocket. La separación es un invariante arquitectónico no negociable del proyecto.

React 19 + Vite 8 + React Flow 12 Node.js 22 + Express 5 + Socket.io Python 3.12 + FastAPI + LangChain Supabase PostgreSQL 16
□

Diagrama de capas

frontend/src · backend/src/index.ts · agent/main.py
Invariante arquitectónico — vision-global.md §1
Tres procesos independientes: Frontend (React+Vite) ↔ Backend (Node+Express+Socket.io) ↔ Agente IA (Python+FastAPI). Comunicación siempre por HTTP/WS. Nunca child_process Node→Python ni fusión de capas.

Backend — backend/src/

AspectoDetalle
StackNode.js 22 · Express 5 · Socket.io
Archivo de entradabackend/src/index.ts — arranca Express en :3001, instancia socket.io, monta /diagrams (CRUD) y llama a attachAgentHandlers(io).
Handlers de socketbackend/src/socketHandlers.ts — autenticación de handshake (JWT Supabase), frescura de token por socket (tokenExp), rate limit por identidad (userId o IP), caché de generaciones, reenvío de eventos del agente al cliente.
Control de admisiónRate limiter en rateLimit.ts, caché global en cache.ts. Estos módulos viven en el backend, no en el agente: el agente queda solo con lógica de IA.
PersistenciaCRUD REST en /diagrams (diagrams.ts). El JWT del usuario se reenvía a Supabase para respetar RLS (auth.uid()). service_role solo para la caché global (generation_cache).
StreamingagentStream.ts — consume el NDJSON del agente vía HTTP y retransmite cada evento JSON al socket del cliente.
Antipatrón prohibido
El backend nunca contiene lógica del agente: no clasifica diagramas, no valida semántica, no orquesta tools. Rate limit y caché son control de admisión, no razonamiento de IA.

Agente IA — agent/

AspectoDetalle
StackPython 3.12 · FastAPI · LangGraph · LangChain · Pydantic
Punto de entradaagent/main.py — servidor FastAPI en :8000 con tres endpoints: POST /generate/stream, POST /refine/stream, POST /refine/resume. También GET /health.
Generación/generate/stream corre el grafo LangGraph de generación y devuelve NDJSON. El resultado pasa por classify_outcome (outcome.py): tres clases de error + done. Parámetro opcional diagram_type para tipo preseleccionado.
Refinamiento/refine/stream construye un DiagramWorkspace desde el diagrama compacto enviado por el backend y corre el loop ReAct con build_agent_graph. Streaming con astream(stream_mode="updates").
Clarificaciones/refine/resume reanuda el grafo pausado por interrupt() usando Command(resume=answer) y el InMemorySaver (estado entre peticiones: excepción consciente al stateless, con migración a Postgres pendiente).
LLM profileVariable LLM_PROFILE en agent/.env: local → Ollama qwen3:1.7b/qwen3:8b · openai → gpt-4o-mini/gpt-4o · anthropic → Claude. Imports lazy por perfil.
Schema-firstTodo lo que cruza un proceso pasa por Pydantic: GenerateRequest, RefineRequest, ResumeRequest, DiagramNode, DiagramSchema, CompactDiagram.

Supabase — única base de datos

PostgreSQL 16. Solo accesible desde el backend, nunca desde el frontend directamente. Dos patrones de acceso:

  • Diagramas de usuario (diagrams): cliente Supabase por petición con el JWT del usuario — auth.uid() impone la propiedad vía RLS.
  • Caché de generaciones (generation_cache): service_role del backend — RLS activa pero sin políticas, solo el backend puede leer/escribir.
Por qué no NoSQL
Nodos y aristas necesitan constraints relacionales (integridad referencial, RLS por usuario). Un documento sin esquema no puede expresar esas garantías.

Por qué nunca se fusionan

razones técnicas y de diseño que cierran la discusión

Tabla de incompatibilidades

Fusión hipotéticaPor qué rompeDecisión cerrada
Frontend + Backend SSR (Next.js) rompe Canvas API y React Flow (requieren DOM de cliente). Un bundle único acoplaría los ciclos de despliegue. React + Vite (no Next.js)
Backend + Agente child_process Node→Python bloquea el event loop y oculta errores. La interfaz HTTP fuerza contratos explícitos (Pydantic / TypeScript) y permite escalar el agente de forma independiente. Agente en Python separado
Agente embebido en Frontend Expone el LLM_PROFILE, las claves de API y los prompts al cliente. Imposible en un navegador sin servidor. —
Frontend → Supabase directo Acoplamiento directo del cliente al esquema de BD; imposible centralizar auth, RLS ni caché. El backend es el único punto de entrada. Backend único punto de acceso a datos

Principio de responsabilidad única por proceso

Cada proceso tiene una única razón de cambio:

  • Frontend — cambia cuando cambia la UX (canvas, panel de chat, imports/exports). No depende de qué LLM se usa.
  • Backend — cambia cuando cambia la política de acceso (auth, rate limit, caché) o el contrato de persistencia. No conoce la lógica del grafo LangGraph.
  • Agente IA — cambia cuando cambia la lógica de generación, el prompt, el modelo o la topología del grafo. No conoce si el cliente usa WebSocket o REST.
Principio — vision-global.md §2
Vertical E2E sobre horizontal: cada sprint atraviesa las tres capas con un caso mínimo, en vez de completar una capa entera. La separación hace posible iterar en paralelo.

Antipatrones activamente bloqueados en revisión

Bloquear en code review
  • child_process Node → Python (cualquier forma)
  • Next.js / cualquier SSR que ejecute código React en servidor
  • Mermaid como pipeline editable (unidireccional, sin modelo de datos)
  • NoSQL para nodos/aristas (sin constraints relacionales)
  • LLM devolviendo strings sin schema (sin Pydantic/Zod en el borde)
  • Lógica del agente en el backend (clasificar, validar diagramas, orquestar tools)
  • Estado de diagrama duplicado entre frontend y backend sin fuente de verdad
  • any en TypeScript · dict sin tipar en Python en código que cruza procesos

Contratos entre procesos

schema-first — todo lo que cruza un proceso tiene schema explícito
Todo lo que cruza un proceso pasa por un esquema. Pydantic en el agente, tipos TS derivados de Zod en backend/frontend. El schema es la fuente de verdad del contrato. — vision-global.md §1.4
