Sistema · Operación

Despliegue

Cómo se levanta MydIAgram. Los tres procesos son artefactos desplegables por separado que se comunican por HTTP y WebSocket; ninguno invoca a otro dentro del mismo runtime. Hay dos caminos: arranque en desarrollo (tres procesos a mano) y Docker Compose (un contenedor por proceso). La base de datos vive en Supabase.

⚙

Topología

Un contenedor por proceso; la base de datos es externa

El agente solo publica su puerto en la red interna de Compose (expose, no ports): nunca es alcanzable desde el navegador. El backend lo localiza por el DNS de servicios (AGENT_BASE_URL: http://agent:8000). Supabase no se contineriza; se apunta por variables de entorno.

Arranque en desarrollo

Base de datos local + tres procesos a mano

1Base de datos local con la CLI de Supabase

Arranca el stack local de Supabase y aplica todas las migraciones de supabase/migrations/ (esquema, RLS, caché y versiones):

supabase start
supabase db reset   # aplica todas las migraciones

Queda escuchando en http://127.0.0.1:54321.

2Instalar dependencias

# agente Python
cd agent && pip install -r requirements.txt && cd ..
# backend y frontend
cd backend && npm install && cd ..
cd frontend && npm install && cd ..

Si vas a usar el perfil local, descarga el modelo de Ollama (ver perfiles de LLM):

ollama pull qwen3:8b

3Levantar los tres procesos (una terminal cada uno)

# terminal 1 — agente IA (:8000)
cd agent && uvicorn main:app --host 0.0.0.0 --port 8000 --reload

# terminal 2 — backend / gateway (:3001)
cd backend && npm run dev

# terminal 3 — frontend (:5173)
cd frontend && npm run dev
Listo

La aplicación queda disponible en http://localhost:5173. El frontend habla con el backend (:3001) por WebSocket, y el backend reenvía al agente (:8000) por HTTP.

▧

Docker Compose

Un servicio por proceso, contra Supabase cloud

docker-compose.yml orquesta los tres procesos como servicios aislados. La base de datos es Supabase en la nube (no se contineriza): se apunta por variables de entorno.

# 1) copia la plantilla y rellena las claves
cp .env.example .env
#    rellena SUPABASE_*, VITE_* e INTERNAL_PROXY_SECRET,
#    y el perfil de LLM en agent/.env

# 2) construye y levanta
docker compose up --build
ServicioPublicaciónNotas
agentexpose: 8000Solo red interna de Compose; nunca se publica al host. Lee su perfil de LLM de agent/.env.
backendports: 3001:3001depends_on: agent. Alcanza al agente por DNS interno: AGENT_BASE_URL: http://agent:8000.
frontendports: 8080:80Estáticos servidos por nginx. depends_on: backend.
Build-time

Las variables VITE_* se hornean en el bundle durante el build del frontend (se pasan como build args en Compose), por lo que deben estar fijadas antes de construir la imagen. Apuntan al backend tal y como lo ve el navegador, no la red interna.

◉

Perfiles de LLM

Un motor por configuración, sin recompilar

El agente es agnóstico al proveedor: ambos perfiles exponen los mismos nombres lógicos (fast y capable), así que el resto del código no cambia. Tras editar el perfil basta reiniciar el microservicio.

local — Ollama (por defecto)

  • Requiere Ollama en marcha y OLLAMA_URL apuntando a su endpoint de chat.
  • Ambos roles apuntan por defecto a qwen3:8b; qwen3:1.7b sirve para el rol rápido a costa de más fallos de formato.
  • La inferencia ocurre en la máquina del operador (o del usuario, en transporte de navegador). Sin coste por token.

openai · anthropic · gemini

  • Requieren la clave del proveedor (OPENAI_API_KEY, ANTHROPIC_API_KEY, GEMINI_API_KEY).
  • Modelos por defecto en agent/.env.example (por ejemplo gpt-4o-mini / gpt-4o).
  • Más calidad y velocidad; la generación ocurre en los servidores del proveedor.
Sin Structured Outputs

La generación llama al modelo con HTTP crudo (httpx), sin response_format: el JSON se emite como texto libre, se parsea con ijson a medida que llega y se valida con Pydantic más un bucle de reintento. La razón está en Modelo de datos y el grafo de generación.
