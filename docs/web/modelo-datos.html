Schema-first: una única fuente de verdad

agent/schemas.py · frontend/src/types.ts

El contrato de datos lo define una sola vez Pydantic en el agente Python. El frontend no repite la estructura con interfaces TypeScript sueltas; en su lugar la declara con schemas Zod y deriva los tipos con z.infer<>. Esto garantiza que tipo estático y validación en runtime no puedan desincronizarse.

El principio queda documentado en el propio código fuente (types.ts línea 53): "tipos en los bordes" — cualquier dato que cruce un límite de proceso (JSON del agente → import del usuario → respuesta del servidor) se valida en runtime; los tipos puramente de UI siguen siendo type/interface sin validador.

Decisión Los schemas Zod son la única fuente de verdad en el frontend. Los tipos TypeScript se derivan del schema (z.infer), no se declaran por separado. Cualquier campo añadido a Pydantic debe replicarse en el schema Zod para que el contrato no se rompa silenciosamente en compilación.

DiagramSchema y CompactDiagram

El modelo raíz DiagramSchema (Pydantic) / diagramSchema (Zod) ensambla title + diagram_type + nodes + edges, más dos campos opcionales y específicos de un tipo que no contaminan al resto: fragments (fragmentos combinados, solo secuencia) y group_layout (geometría manual de contenedores de grupo, solo arquitectura/casos de uso). Ambos se detallan en la sección Variación del JSON por tipo. El campo title es human-facing; el agente lo proporciona pero el frontend lo stripea al refinar:

ALLOWED_* por tipo de diagrama

agent/schemas.py — ALLOWED_NODE_TYPES · ALLOWED_EDGE_TYPES

Los diccionarios ALLOWED_NODE_TYPES y ALLOWED_EDGE_TYPES son la fuente de verdad de qué tipos semánticos son válidos para cada tipo de diagrama. Son el correlato-datos de la guía que prompts.py da al LLM: el prompt dice qué usar; estos mapas validan que se haya usado. Un elemento puede ser Pydantic-válido (valor dentro del enum global) y semánticamente inválido para su diagrama (p. ej. edge_type=one_to_many en un flowchart).

Diseño — Registro parcial deliberado Un DiagramType sin entrada en ALLOWED_* no se valida semánticamente: la función devuelve True para cualquier tipo del enum (fallback permisivo). Si se añade un nuevo tipo y se olvida registrarlo, el sistema degrada a "solo Pydantic" en vez de rechazar todos sus diagramas. El principio rector es: fallar hacia permisivo, no hacia roto.

API de consulta

Dos funciones exportadas en schemas.py encapsulan la consulta a estos diccionarios:

  • node_type_allowed(diagram_type, node_type) → bool — Devuelve True si el tipo de nodo es válido para ese diagrama, o si el diagrama no está registrado (fallback permisivo).
  • edge_type_allowed(diagram_type, edge_type) → bool — Ídem para aristas.

Variación del JSON por tipo de diagrama

Misma forma, distinto contenido · agent/prompts.py · frontend/src/ui/utils

El esquema es uniforme: todos los tipos de diagrama usan el mismo DiagramNode y el mismo DiagramEdge. No hay un campo shape, kind o variant por nodo que cambie la forma del JSON según el diagrama. Lo que varía es el contenido, en tres ejes:

  1. Qué valores son válidos en node_type / edge_type — lo regula ALLOWED_* (sección anterior). El subtipo dentro de un diagrama también se distingue por node_type (en un flowchart: step vs decision vs terminator), no por un campo aparte.
  2. Qué se mete en attributes — un array de strings con convenciones de prefijo distintas según el tipo. Aquí está casi toda la diferencia «pequeña» que se percibe entre diagramas.
  3. Campos top-level extra que solo usa un tipo concreto: fragments (secuencia) y group_layout (arquitectura / casos de uso).
Clave attributes es siempre list[str]: nunca objetos anidados. La riqueza (tipo de columna, PK/FK, tecnología, agrupación) se codifica como convención dentro del string. Un ERD no tiene un campo estructurado de columnas: lleva ["id: INT PK", …] y es el frontend quien interpreta ese texto (p. ej. TableNode detecta PK/FK por substring).

Campos extra opcionales — fragments y group_layout

fragments (solo secuencia). Un fragmento combinado UML (alt/opt/loop/par) no es ni nodo ni arista: es una tercera entidad que envuelve un rango ordenado de mensajes (ids de edges) con semántica de control de flujo. Por eso vive en su propio array top-level y no en attributes. alt tiene ≥2 operandos (if/else, cada uno con su guarda); el resto, uno. Anidan por referencia explícita (child_fragment_ids), no por contención. Se validan referencialmente (cada message_id / hijo debe existir); un fragmento inválido se descarta en limpio, no se reintenta como nodos/aristas.

"fragments": [
  {"id": "f1", "kind": "alt", "operands": [
    {"guard": "[usuario válido]", "message_ids": ["e3"], "child_fragment_ids": []},
    {"guard": "[else]",          "message_ids": ["e4"], "child_fragment_ids": []}
  ]}
]

group_layout (arquitectura / casos de uso). Geometría manual {x, y, width, height} de los contenedores de grupo, indexada por el id derivado del contenedor (group__<Nombre> en arquitectura, el id del system en casos de uso). Es solo-canvas: si el usuario mueve o redimensiona un contenedor su geometría se persiste aquí, pero diagramToJson lo stripea antes de enviar al agente (que solo recibe diagram_type/nodes/edges/fragments).

Validación en tres niveles

Local · Referencial · Estructural

El sistema aplica tres capas de validación secuenciales. Cada capa detecta una clase distinta de error; su combinación garantiza que el diagrama que llega al frontend es Pydantic-válido, semánticamente consistente, referencialmente íntegro y estructuralmente correcto para su tipo.

Nivel 1 — Validación local (por pieza)

Cada nodo y arista se valida individualmente contra los schemas Pydantic/Zod. La validación comprueba dos cosas: (a) que los valores estén dentro del enum global (NodeType, EdgeType), y (b) que sean semánticamente válidos para el diagram_type del diagrama (consulta a ALLOWED_NODE_TYPES / ALLOWED_EDGE_TYPES). Un fallo en este nivel dispara un ValidationError de Pydantic, que el bucle de reintento del agente utiliza para corregir al LLM.

Nivel 2 — Validación referencial (integridad de grafo)

Una arista cuyo source o target no exista entre los nodos del diagrama es una arista huérfana. El agente la detecta en extract_edges y reintenta la extracción; el frontend la rechaza al importar un JSON externo mediante diagramImportSchema (Zod .refine()):

diagramImportSchema = diagramSchema.refine(
  (d) => {
    const ids = new Set(d.nodes.map((n) => n.id));
    return d.edges.every((e) => ids.has(e.source) && ids.has(e.target));
  },
  { message: "El diagrama tiene aristas que referencian nodos inexistentes (huérfanas)" },
)

Nivel 3 — Validación estructural (conjunto)

Implementada en structural.py por validate_structure(diagram_type, nodes, edges). Un diagrama puede tener todos sus elementos válidos por separado y aun así ser estructuralmente incorrecto: un flowchart sin terminators, un mindmap sin raíz, una secuencia con un solo actor. Cuando la validación estructural falla, genera uno o varios StructuralGap {type, reason} que el LangGraph usa para decidir desde dónde relanzar el bucle de relleno:

  • type = "nodes" — falta un nodo; el bucle relanza desde extract_nodes.
  • type = "edges" — los nodos existen pero falta conectarlos; el bucle relanza desde extract_edges.
Principio rector Validar el mínimo que hace a un diagrama ser de su tipo; nunca más. Por eso erd, architecture y use_case son permisivos a propósito en el nivel estructural: una tabla sin FK, un servicio sin cablear o un actor sin casos asociados son legítimos en esos tipos.

Taxonomía de desenlaces (outcomes)

agent/outcome.py · classify_outcome()

La función classify_outcome(state, *, crashed) en outcome.py es una función pura — no es un nodo del grafo — que traduce el estado final del LangGraph en el evento terminal del stream (done o error). Su justificación: hay dos desenlaces que ningún nodo de salida puede observar:

  • guard-reject: el grafo sale por END justo tras guard, saltándose cualquier nodo terminal.
  • excepción interna: si un nodo revienta, ainvoke lanza y no llega a ejecutarse ningún nodo de salida.

main.py es el único punto que observa los tres desenlaces posibles (estado limpio, guard-reject y crash), de ahí que la clasificación resida aquí y no dentro del grafo.

Frontera done / error

El umbral es ≥1 nodo válido. Con al menos un nodo el diagrama es utilizable → done (degradado o no). Con cero nodos no hay nada que mostrar → error. Guard-reject y excepción son sendas categorías de error.

Tabla de categorías de error

CategoríaCausaMensaje accionable
internal_error Excepción interna (crashed=True o state is None) Error interno; reintentar en unos segundos
not_a_diagram Guard-reject: is_diagram_request = False El texto no describe un sistema representable; reformular indicando entidades y relaciones
empty Fallo total: diagram.nodes = [] tras el bucle de relleno No se extrajo ningún elemento; describir con más detalle qué elementos lo componen

Degradación parcial — done + degraded: true

Cuando el diagrama es utilizable pero incompleto (el agente agotó los reintentos sin cumplir todas las reglas estructurales), el evento done incluye degraded: true y la lista degradations: Degradation[]. El frontend compone el aviso al usuario por categoría ('nodes' | 'edges' | 'structure'). En el agente, las degradaciones se acumulan en el campo degradations del DiagramState por los branches de rendición.

Decisión — Snapshot completo en done El evento done lleva el snapshot completo del diagrama también en el flujo de generación (no solo en refinamiento). Los eventos incrementales node_ready / edge_ready no transmiten el diagram_type, que el frontend necesita para poder refinar después (CompactDiagram lo exige; sin él, /refine/stream da 422). El snapshot también reconcilia cualquier evento incremental perdido. Patrón uniforme: eventos = UX efímera; done = verdad.
