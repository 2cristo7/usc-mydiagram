Grafo ReAct de Refinamiento

El agente conversacional que edita diagramas existentes vía tool calls: loop agent ↔ tools, fin por ausencia de tool_calls, rama regenerate como escape hatch y pausa real clarify con interrupt() de LangGraph.

Visión general del grafo

agent/agent_graph.py · build_agent_graph()

El grafo de refinamiento implementa el patrón ReAct (Reason + Act) sobre LangGraph. A diferencia del grafo de generación, que es un pipeline lineal append-only, este grafo es un loop controlado: el LLM razona sobre el diagrama actual y decide qué tool invocar; el resultado de la tool (observación) se añade al historial y el LLM decide de nuevo, hasta que responde sin llamar a ninguna tool.

El loop termina cuando el LLM produce una AIMessage sin tool_calls — no existe un marcador explícito de fin. tools_condition de LangGraph inspecciona el último mensaje y enruta a END si no hay calls pendientes.

El grafo se construye por petición (build_agent_graph(ws)): las tools cierran (closure) sobre el DiagramWorkspace de esa petición concreta. Dos refinamientos concurrentes compilan grafos distintos y nunca comparten el mismo workspace. (Decisión en docs/vision-global.md §3.)

Topología completa del grafo

Decisión
El grafo se compila por petición (build_agent_graph(ws)) porque las tools cierran sobre el DiagramWorkspace concreto. Un ToolNode global mutaría un diagrama compartido y dos refinamientos concurrentes se pisarían. Ver vision-global.md §3.

¿Qué contexto recibe el agente? El diagrama actual entero

El refinamiento no parte de cero: el LLM ve todo el diagrama antes de editarlo

Para refinar ("añade un atributo a Usuario", "borra esa relación", "conecta A con B") el agente necesita saber qué hay ya en el lienzo. Por eso, en cada petición de refinamiento se le envía el diagrama actual completo —todos los nodos y todas las aristas— serializado como JSON e inyectado en el system prompt del LLM. No es un resumen, no es una muestra, y no es solo el texto del usuario: es el diagrama entero.

El prompt del usuario viaja en el HumanMessage; el diagrama actual viaja en el SystemMessage (build_system_prompt(ws)). Esa separación es deliberada: la instrucción es lo que cambia turno a turno; el diagrama es el contexto sobre el que se razona.

De dónde sale ese contexto: el recorrido del diagrama

CapaQué hace con el diagramaArchivo · línea
Frontend Al detectar que ya hay diagrama en el lienzo emite message:refine con { prompt, diagram }. El diagram es el JSON compacto: todos los nodos (sin position), todas las aristas (sin data visual), diagram_type y fragments si los hay. useWebSocket.ts:771 · diagramToJson.ts:27
Backend Reenvía { prompt, diagram } tal cual a POST /refine/stream. No interpreta ni reduce el diagrama; por eso este endpoint no se cachea (la respuesta depende del diagrama de entrada). socketHandlers.ts:397
Agente Reconstruye el DiagramWorkspace desde el JSON, serializa el diagrama entero con ws.to_compact() y lo incrusta en el system prompt. main.py:403 · agent_graph.py:193

Qué incluye —y qué se omite— en el JSON que ve el LLM

El diagrama se envía como CompactDiagram (ws.to_compact()), una vista semántica del diagrama: lo que el agente necesita para editar, sin el ruido visual que solo importa al render.

CamposPor qué
Se envía Por nodo: id, label, node_type, attributes. Por arista: id, source, target, label, edge_type. Más diagram_type y fragments (secuencia). El agente edita por id (update_node, delete_edge…); necesita ver los id y las conexiones existentes.
Se omite position de los nodos, data visual de las aristas (waypoints, forma, flechas) y el title del diagrama. Es información de presentación. El layout lo recalcula dagre en el frontend en cada render; el title es human-facing. Mandarlo solo gastaría tokens.
El contexto es el snapshot inicial; los cambios se siguen por las observaciones
El system prompt se construye una vez al arrancar el refinamiento, con el diagrama tal y como llegó. Durante el loop ReAct el agente no recibe un system prompt regenerado en cada turno: ve ese snapshot inicial más las observaciones que devuelven sus propias tools ({"id": …}, {"ok": true, …}), que le confirman cada mutación. La fuente de verdad final es el ws mutado, que se sirve completo en el evento done.
Implicación de escala
Como el diagrama entero entra en el prompt, su tamaño crece con el nº de nodos/aristas: diagramas muy grandes consumen más contexto por turno. La vista compacta (sin position ni data) es precisamente la primera defensa contra ese coste. Una poda selectiva del contexto (enviar solo el subgrafo afectado) sería una optimización futura, no el comportamiento actual: hoy se envía todo.

Nodos del grafo

StateGraph(MessagesState) · cuatro nodos + entry point

Nodo Tipo Responsabilidad Entra por Sale a
agent async fn Invoca chat_model.ainvoke(messages) y devuelve la respuesta al estado. Si el LLM incluye tool_calls, tools_condition enruta a tools; si no, a END. START, tools (vía route_after_tools), clarify (tras resume) tools_condition → tools o END

clarify async fn · interrupt Pausa real: llama a interrupt({"question":…, "options":…}). El grafo se congela (checkpointer) y el stream devuelve la pregunta. Al reanudar, inyecta la respuesta como HumanMessage y vuelve a agent. tools (vía route_after_tools) agent

Estado del grafo: MessagesState

El grafo usa MessagesState de LangGraph como estado: una lista de mensajes (HumanMessage, AIMessage, ToolMessage…) con reducer add_messages que acumula en vez de reemplazar. El historial completo es la fuente de verdad; extract_history() lo posprocesa al final para derivar refinement_history sin mantener una segunda lista en paralelo.

Principio de diseño
Derivar al final, no construir en paralelo. extract_history() es una función pura sobre messages: empareja cada AIMessage.tool_calls con su ToolMessage por tool_call_id. Mismo patrón que classify_outcome: una segunda fuente de verdad que crece en paralelo puede desincronizarse; derivar al final es siempre consistente y testeable sin el grafo.

Condiciones de routing

tools_condition · route_after_tools · _last_marker()

tools_condition (LangGraph prebuilt)

Inspecciona el último mensaje del estado. Si es una AIMessage con tool_calls no vacíos → "tools". Si no → END. Esta condición es la que hace que el loop termine naturalmente: el LLM responde sin pedir tools cuando considera que ha completado la tarea.

route_after_tools (custom)

Se ejecuta tras ToolNode. Inspecciona los ToolMessage del último turno (solo los del turno actual, no del historial completo) buscando marcadores especiales en las observaciones JSON:

Marcador detectadoRutaPrioridad
{"_interrupt": "clarification", …} "clarify" 1ª (mayor)
{"_regenerate": true, …} "regenerate" 2ª
Sin marcadores "agent" fallback
Orden importa
Si en el mismo turno el LLM llamó a ask_clarification y a regenerate_from_scratch (improbable pero posible), la clarificación tiene prioridad: preguntar antes de tirar el diagrama.

_last_marker(messages, key)

Helper puro que recorre los mensajes desde el final mientras encuentre ToolMessage. El primer mensaje que no sea ToolMessage corta la búsqueda. Así solo inspecciona las observaciones del turno actual, no las de turnos anteriores donde el mismo marcador pudo aparecer.

Escape hatch: regenerate_from_scratch

Cuando el cambio pedido no es incremental

La mayoría de las tools editan el workspace incrementalmente: add_node, update_node, delete_edge… Pero algunos cambios no son incrementales: cambiar el tipo de diagrama, rehacer el diagrama entero con otra semántica o corregir una topología muy errónea.

Para estos casos existe regenerate_from_scratch, que el LLM puede llamar como cualquier otra tool pero cuya ejecución en el grafo es radicalmente distinta:

  • La tool regenerate_from_scratch(prompt, diagram_type?) en DiagramWorkspace no hace nada: devuelve el marcador {"_regenerate": true, "prompt": "...", ...} como observación. La frontera tool/grafo es explícita.
  • El nodo regenerate del grafo lee el marcador con _last_marker(), extrae el prompt y corre build_graph().ainvoke(initial_generation_state(prompt)).
  • El resultado reemplaza ws.diagram_type, ws.nodes y ws.edges.
  • El nodo sale a END directamente: no vuelve al loop del agente. El endpoint /refine/stream sirve el workspace actualizado uniforme con cualquier otro refinamiento.
Decisión
regenerate_from_scratch sale por rama dedicada (no es tool incremental) porque produce un diagrama entero, no una observación pequeña. Volver al agente con el diagrama nuevo como contexto implicaría una llamada LLM extra innecesaria. La tool es "intención + datos"; el nodo del grafo es la ejecución.

Interrupt / Resume: ask_clarification

LangGraph interrupt() · checkpointer InMemorySaver

Cuando el LLM detecta que la instrucción del usuario es ambigua puede llamar a ask_clarification(question, options?). La tool devuelve el marcador _interrupt; el nodo clarify llama a interrupt() de LangGraph, que congela el estado del grafo hasta que el cliente reanude con la respuesta.

Diseño del nodo clarify: libre de efectos

El nodo clarify es deliberadamente libre de efectos: solo llama a interrupt() y no ejecuta nada más. Esta decisión es crítica por cómo funciona LangGraph al reanudar:

  • Al reanudar, LangGraph re-ejecuta el nodo interrumpido desde su inicio.
  • Las tools hermanas del mismo turno (p. ej. un add_node que llegó antes de ask_clarification) ya corrieron en ToolNode y están checkpointeadas: no se repiten.
  • Si el interrupt() viviera en ToolNode junto con esas tools, todas se ejecutarían dos veces al reanudar (duplicado silencioso).
  • En un nodo aparte, la segunda ejecución de interrupt() es idempotente: devuelve la respuesta del usuario en vez de pausar.
Decisión
Sin validador LLM previo al reanudar: interpretar la respuesta del usuario (¿contesta a la pregunta o pide algo distinto?) es trabajo del propio agente en el loop. Un validador extra duplicaría la llamada LLM y sería una segunda fuente de interpretación con menos contexto. Estado entre pausa y reanudación en InMemorySaver: excepción consciente al stateless (migrará a checkpointer Postgres).

Streaming de tool calls

tool_events() · astream(stream_mode="updates")

El endpoint /refine/stream usa astream(stream_mode="updates") de LangGraph para emitir eventos NDJSON a medida que el grafo avanza. La función tool_events(update, ws) traduce cada update a la lista de eventos correspondiente.

Update (nodo del grafo)Mensaje detectadoEvento NDJSON emitido
agent AIMessage con tool_calls {"_type":"tool_call", "id":…, "tool":…, "args":…}
tools ToolMessage {"_type":"tool_result", "id":…, "tool":…, "result":…, [delta]}
clarify — No emite (evento clarification propio)
regenerate — No emite (lleva el diagrama en done)

Delta del servidor

Para add_node, update_node y add_edge, el evento tool_result adjunta la pieza completa leída del workspace (no solo el id del resultado). El cliente la aplica literal, sin reimplementar semántica del servidor (slugs, methods→attributes, cascade de borrados, etc.).

Reconciliación incondicional
Los eventos de streaming son UX efímera: retroalimentación visual mientras el agente trabaja. El evento done lleva el snapshot completo del diagrama + refinement_history y se aplica siempre, independientemente de los eventos parciales recibidos. Si un evento se perdió o llegó desordenado, el done corrige. La guarda de idempotencia del render solo puede fallar "hacia aplicar de más", nunca hacia no-reparar.

¿Por qué no astream_events?

astream_events ofrece granularidad sub-nodo (tokens individuales), pero el mapeo nodo→evento ya es 1:1 con la topología del grafo. Usar stream_mode="updates" evita filtrar el 95% de eventos irrelevantes (tokens de texto) y la función tool_events() es pura y testeable sin grafo ni LLM.

Las 9 tools del agente

agent/tools.py · DiagramWorkspace · build_tools(ws)

Las tools son lógica pura sobre el DiagramWorkspace: testeables sin LLM ni grafo. Cada tool devuelve una observación como JSON-string. En éxito, los datos que el plan pide ({"id":…}, lista, etc.); en error, {"error": "<mensaje accionable>"} que el agente puede leer y autocorregir.

ToolArgs principalesObservación en éxitoNotas
find_node query: str [{id, label, type}, …] Substring primero; fuzzy con difflib si no hay match. Usar antes de cualquier ref por id.
add_node node_type, label, attributes?, methods? {"id": slug} Valida node_type contra el tipo de diagrama. El nodo nace desconectado. methods se anexan a attributes.
update_node id, label?, node_type?, attributes? {"ok": true, "id": …} Modificación parcial. attributes reemplaza la lista completa si se da.
delete_node id: str {"ok": true, "deleted_node": id, "deleted_edges": […]} Cascade: borra todas las aristas conectadas. Las devuelve en la observación.
add_edge source, target, edge_type, label? {"id": slug} Nunca crea nodos. Si falta un extremo, error accionable. Valida edge_type.
delete_edge id? | source + target {"ok": true, "deleted_edge": id} Acepta id O source+target (cualquier dirección). Si hay múltiples aristas entre los mismos nodos, devuelve candidatos.
apply_layout — {"ok": true, "note": "…"} No-op en servidor: dagre recalcula en el frontend en cada render. Tool de señal semántica.
ask_clarification question: str, options?: list[str] {"_interrupt": "clarification", "question": …, "options": […]} Solo señal. La pausa real es interrupt() en el nodo clarify del grafo.
regenerate_from_scratch prompt: str, diagram_type?: DiagramType {"_regenerate": true, "prompt": …, "diagram_type": …|null} Solo señal. La ejecución del pipeline de generación es el nodo regenerate del grafo.

Frontera lógica pura / grafo

Las dos últimas tools (ask_clarification y regenerate_from_scratch) siguen el patrón intención + datos en la tool, ejecución en el grafo. Esta frontera es explícita en el código:

tools.py: lógica pura, sin LangGraph, sin imports del grafo.
agent_graph.py: ensambla el grafo, lee los marcadores y ejecuta el efecto real (interrupt(), build_graph().ainvoke()).

El DiagramWorkspace es deliberadamente mutable (add/update/delete legítimos en el pipeline de refinamiento, a diferencia del pipeline de generación que es append-only). Dos refinamientos concurrentes reciben instancias distintas del workspace, no comparten estado.

Sistema de prompts por tipo de diagrama

El system prompt del agente (build_system_prompt(ws)) se genera por petición e incluye:

  • El diagrama actual serializado como JSON compacto (ws.to_compact()).
  • Los node_type y edge_type válidos para el tipo de diagrama, extraídos de ALLOWED_NODE_TYPES / ALLOWED_EDGE_TYPES (misma fuente de verdad que la validación de tipos).
  • Una guía específica para diagramas de secuencia (_SEQUENCE_AGENT_GUIDE) que aclara que los participantes son nodos y las interacciones son aristas, evitando el error más frecuente: crear un nodo "Handshake inicial" en vez de tres aristas SYN/SYN-ACK/ACK.

Modelo de chat y perfiles LLM

agent/llm.py · get_chat_model("capable") · tres perfiles

El agente ReAct requiere un BaseChatModel de LangChain con soporte a bind_tools(). La función get_chat_model(profile) enruta según LLM_PROFILE en el entorno:

LLM_PROFILEPerfil "capable"Clase LangChain
local (defecto) Ollama qwen3:8b ChatOllama
openai gpt-4o ChatOpenAI
anthropic claude-3-5-sonnet-* ChatAnthropic
Dos APIs coexisten
El pipeline de generación usa call_llm/stream_llm (HTTP crudo con httpx, una completion, texto plano). El agente ReAct usa get_chat_model() (chat-model de LangChain con bind_tools()). Ambas enrutan por LLM_PROFILE. Migrar toda la generación a chat-models sería un refactor masivo sin ganancia; las dos abstracciones coexisten por necesidad. Decisión cerrada en vision-global.md §3.
