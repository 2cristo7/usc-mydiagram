Stack base e infraestructura

D01–D05 · Cimientos no negociables

Decisión D01

Supabase (PostgreSQL) como única BD. Los nodos y aristas de un diagrama son entidades con identidad, relaciones y constraints de integridad referencial que solo un modelo relacional garantiza de forma nativa. Un store NoSQL como Mongo o Firestore haría recaer esas constraints en el código de aplicación, con el riesgo de aristas huérfanas persistentes.

Decisión D02

Proceso Python independiente para el agente IA. LangChain, LangGraph y el ecosistema de herramientas de IA (transformers, ijson, httpx para LLM) son de facto Python. Ejecutarlos desde Node.js vía child_process implicaría serializar el estado completo del agente en cada llamada, impide streaming granular y acopla los ciclos de vida de los dos procesos. La arquitectura de tres procesos independientes (frontend / backend / agente) con HTTP/WS entre ellos es invariante del proyecto.

Decisión D03

React 19 + Vite 8 (no Next.js ni ningún framework con SSR). El canvas de diagramas usa la Canvas API y React Flow, ambas client-only: requieren acceso al DOM y a eventos de puntero que no existen en el servidor. Next.js SSR renderizaría los componentes en servidor y los hidrataria en cliente, produciendo mismatches de hidratación insalvables. Vite en modo SPA da HMR sub-segundo y bundles óptimos sin ninguna complejidad de SSR.

Decisión D04

Express 5 + Socket.io como backend. El streaming de nodos y aristas desde el agente requiere un canal bidireccional. Socket.io sobre Express es la opción con mejor soporte y documentación para WS en Node.js. Fastify o Nest habrían añadido abstracción sin ganancia real para un servidor cuya responsabilidad central es enrutar HTTP → Python y mantener el socket → cliente.

Decisión D05

React Flow 12 (no D3 ni cytoscape). React Flow expone un modelo declarativo de nodos y aristas como objetos React, con edición interactiva (drag, resize, conexiones) integrada. D3 es un motor de renderizado SVG de bajo nivel: habría que reimplementar todo el modelo de interacción. Cytoscape tiene un modelo de datos propio que no se mapa a tipos TS sin conversión constante.

✓

Contratos y schema-first

D06 · Decisión transversal

Decisión D06

Schema-first: Pydantic en el agente, Zod en el frontend; tipos TS derivados con z.infer. Todo lo que cruza el borde de un proceso pasa por un schema en runtime. Los tipos TypeScript se borran en compilación — si el schema y el tipo fueran archivos distintos, podrían desincronizarse silenciosamente. z.infer<typeof schema> hace al schema la única fuente de verdad: tipo y validador son la misma declaración. El import de un .json externo usa diagramImportSchema = diagramSchema.refine(integridad referencial) porque es entrada externa no confiable.

Alcance de la decisión

Los tipos puramente de UI (Message, UIState, ToolTrace*, Degradation…) siguen como type/interface sin Zod. No entran por ningún borde externo — son estado interno del cliente — así que el runtime overhead de un schema no aporta nada.

Pipeline de validación en capas

D07–D15 · Núcleo del agente

El núcleo de la generación es un grafo LangGraph con tres categorías ortogonales de validación, cada una con su propio presupuesto de reintento y canal de estado. La decisión de no usar Structured Outputs (DO) como única garantía es la más debatida del proyecto y merece tratamiento separado.

Decisión revisitada D08

Validación en capas con realimentación en vez de Structured Outputs.

La garantía sintáctica de SO (JSON siempre parseable) no cubre los errores dominantes del dominio: aristas huérfanas, node_type semánticamente inválido para ese DiagramType, flowcharts sin nodo de inicio — todos schema-válidos en Pydantic y en SO. Las capas de validación, necesarias de todos modos para esos errores, también atrapan la sintaxis (Pydantic). Descartar en SO y generar de nuevo es más caro que retener lo válido y realimentar solo lo inválido.

SO queda registrado en docs/pendientes.md como refuerzo opcional: evaluar compatibilidad con el streaming ijson y la paridad de los tres perfiles (local/openai/anthropic).

D09
Streaming progresivo sin retracción
Solo se emite al canvas lo confirmado válido. El reintento opera sobre huérfanas retenidas — nunca streameadas. El resultado es siempre "solo añadir": no hay reconciliación ni flash de elementos que desaparecen.
Invariante: una vez emitido, no se retracta
D10
Prompts específicos por DiagramType
Un prompt genérico con ejemplo ERD producía topologías lineales para todos los tipos — flowcharts sin nodo de decisión, casos de uso sin actores. El registro en agent/prompts.py es además la fuente de verdad para la validación semántica del lado agente.
Fallback genérico para tipos sin entrada registrada
D11
extract_nodes = única fuente de nodos
Una arista huérfana (typo en ID vs. nodo olvidado) es indistinguible desde su propia señal. Dejar que la regeneración de aristas invente nodos reintroduce alucinación. Si los nodos son malos, el usuario regenera desde 0.
Las aristas nunca crean nodos — invariante ortogonal
D12
Tipos permitidos en schemas.py, fallback permisivo
El contrato de qué tipo es válido para qué diagrama vive junto a los enums, no en el módulo de prompts (evita dependencia invertida). Si se añade un tipo y se olvida registrarlo, degrada a solo-Pydantic en vez de rechazar todos sus diagramas.
Fallar hacia permisivo, nunca hacia roto
D13
Fallo estructural = carencia, no pieza inválida
No hay pieza que retener/corregir — falta algo que el LLM nunca generó. Canal structural_gaps (replace) + bucle de relleno aditivo con schema_retry_count propio. operator.add en nodos/aristas: el relleno solo añade, respetando el invariante de streaming.
Presupuestos separados por tipo de fallo
D14
Modo de extractores por CONTENIDO del estado
El contador es presupuesto, no señal de modo. Venir de validate_schema implica que el bucle local ya cerró; un invalid_* residual no debe secuestrar el modo. Prioridad: rellenar > corregir > normal.
Detectar modo por hay-gaps / hay-invalid, no por contador
D15
classify_outcome función pura en outcome.py
Un nodo terminal no puede ver todos los desenlaces: el guard-reject sale por END saltando nodos de salida; la excepción lanza y no termina ningún nodo. main.py es el único punto que ve los tres. Función pura → testable sin grafo.
≥1 nodo válido → done; 0 nodos → error
D07
LangGraph para orquestación del agente
Los bucles de reintento con branches condicionales (validar → si error → regenerar → volver a validar) son nodos y edges del grafo. Sin LangGraph, el control de flujo viviría en código imperativo sin checkpointing ni observabilidad.
Tools → ToolNode nativo en LangGraph

Agente ReAct y refinamiento

D16–D19 · Refinamiento conversacional

Decisión D16

Dos APIs de LLM conviviendo por necesidad técnica. La generación usa call_llm/stream_llm (HTTP crudo a Ollama/OpenAI, una completion, texto). El agente ReAct usa get_chat_model (chat-model de LangChain con bind_tools()). bind_tools()/ToolNode exigen un BaseChatModel que la abstracción httpx cruda no implementa. Migrar toda la generación a chat-models sería un refactor masivo sin ganancia funcional — la generación no necesita tools. Ambas APIs enrutan por el mismo LLM_PROFILE; el agente soporta tres perfiles (local→ChatOllama, openai→ChatOpenAI, anthropic→ChatAnthropic) con imports lazy por perfil.

Decisión D17

Puente tools↔workspace construido POR PETICIÓN (closure). Un ToolNode global con referencias estáticas al workspace mutaría estado compartido entre peticiones concurrentes: dos usuarios refinando al mismo tiempo se pisarían. El closure captura el DiagramWorkspace de esa petición concreta. El contrato que ve el LLM (nombre + args_schema) es estático; la ejecución apunta a esa instancia. El loop ReAct termina por ausencia de tool_calls, no por marcador explícito.

Decisión D18

ask_clarification usa interrupt() en nodo clarify dedicado, libre de efectos. LangGraph re-ejecuta el nodo interrumpido desde su inicio al reanudar. Si el interrupt viviera en ToolNode, las tools hermanas del mismo turno se ejecutarían dos veces (duplicado silencioso). En nodo aparte, las tools hermanas quedan checkpointeadas y el interrupt es idempotente. La reanudación es /refine/resume {thread_id, answer} con Command(resume=...) sin validador LLM previo: interpretar la respuesta (¿contesta o pide otra cosa?) es trabajo del propio agente en su loop.

Estado en memoria — deuda consciente

El estado entre pausa (interrupt()) y reanudación (Command(resume=...)) vive en InMemorySaver + workspace pendiente por thread_id en memoria del proceso. Es una excepción consciente al principio sin estado del backend, con salida planificada: migrar a checkpointer Postgres.

Decisión D19

refinement_history derivada de los messages finales con función pura extract_history. Cada AIMessage.tool_calls se empareja con su ToolMessage por tool_call_id. La traza ya existe en el estado del grafo; una lista paralela construida durante el loop sería una segunda fuente de verdad capaz de desincronizarse — mismo principio que classify_outcome (D15).

Backend: seguridad y admisión

D20–D22 · Acceso a datos, caché y frescura de token

El backend concentra responsabilidades de acceso a datos, rate limiting y caché, además de actuar como pasarela hacia el microservicio de IA. La distinción clave: control de admisión ≠ lógica del agente. El agente sigue siendo el único lugar donde se clasifican diagramas, validan esquemas y orquestan tools.

Decisión D20

El backend es el único punto de acceso a datos de persistencia. El frontend nunca llama a Supabase directamente. El CRUD vive en backend/src/diagrams.ts; el frontend solo conoce los endpoints REST. El JWT del usuario se reenvía a Supabase para que auth.uid() imponga la propiedad mediante RLS — sin bypass de service_role para diagramas de usuario. Esto concentra auth y topología de BD en un único punto (un cambio de esquema solo afecta al backend, no al frontend).

Decisión D21

Caché global de generaciones con service_role, sin user_id, RLS activada pero sin políticas. El namespace de caché no tiene propietario — es compartida entre usuarios — por lo que no tiene user_id. RLS sin políticas bloquea cualquier acceso desde cliente y solo permite service_role, evitando el envenenamiento de la caché compartida desde el cliente. Clave (prompt_key, model): el namespace por modelo impide servir un diagrama Qwen a un usuario de GPT-4o. Solo se cachea éxito limpio (≥1 nodo, no degradado): cachear un diagrama degradado lo serviría como bueno. El botón "Regenerar" fuerza generación saltándose el lookup. checkRateLimit corre ANTES del lookup — un hit cuenta y un usuario en el tope no llega a la caché.

Decisión D22

Frescura del token en sockets vivos por revalidación perezosa anclada al exp. El handshake guarda socket.data.tokenExp. Las 4 entradas que corren el agente comparan now >= exp (enteros, sin cripto ni red) y cortan (auth:expired + disconnect) si caducó. La verificación cripto cara ocurre solo al entrar un token nuevo (auth:refresh). Identidad inmutable en un socket vivo: si llega un sub distinto, se corta. socket.data.tokenExp es estado de la conexión, no de aplicación — no va a BD; el backend sigue sin estado en lo que importa.

Ventana de revocación

Solo se compara el campo exp del JWT, no se consulta Supabase. Una revocación server-side se respeta como muy tarde a la expiración del token (≤1h). Esto es un trade-off explícito: el coste de la verificación cripto en cada llamada no se justifica para una ventana aceptable de 1h.
